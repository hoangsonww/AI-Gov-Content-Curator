## Learnings
1. [KEEP] The Terraform state backend (S3 + DynamoDB) must be created manually before `terraform init` — it's a chicken-and-egg problem. Added a `make bootstrap` target that creates the S3 bucket and DynamoDB table using AWS CLI, then runs `terraform init`. This only needs to run once per AWS account.
2. [KEEP] ECS Fargate task definitions require explicit `executionRoleArn` (for pulling images from ECR and writing CloudWatch logs) and `taskRoleArn` (for the application's own AWS API calls). Confusing these two roles is a common source of "access denied" errors. The execution role needs `ecr:GetAuthorizationToken`, `ecr:BatchGetImage`, `logs:CreateLogStream`, `logs:PutLogEvents`. The task role needs `secretsmanager:GetSecretValue` for API keys.
3. [KEEP] The Kubernetes CronJob `activeDeadlineSeconds: 1800` (30 minutes) is critical for the crawler. Without it, a hung Puppeteer process (e.g. waiting for a page that never loads) keeps the pod alive indefinitely, consuming cluster resources. The deadline ensures the pod is killed and the job marked as failed after 30 minutes regardless of what the crawler code is doing.
4. Istio's VirtualService path matching is prefix-based by default. The rule `/api/*` matches `/api/health` but also `/api-docs` if a trailing-slash-insensitive match is used. Used `exact` matching for specific paths and `prefix` only for `/api/` (with trailing slash) to avoid accidental route capture.
5. Prometheus scrape interval (15s default) generates significant cardinality for per-route metrics. The backend exposes request count and latency per route, which with 15 routes × 5 methods × 3 pods = 225 time series updated every 15 seconds. Acceptable for this scale but would need aggregation or exemplars at higher cardinality.

## Gotchas
- The ECS scheduled task module uses `aws_cloudwatch_event_rule` which requires the cron expression in CloudWatch format, not standard cron. CloudWatch uses 6 fields (including year) and `?` for day-of-month or day-of-week. Standard 5-field cron `0 9 * * *` becomes `cron(0 9 * * ? *)`. This tripped up the first `terraform plan`.
- Kubernetes resource quotas on the namespace (`requests.cpu: 4`, `requests.memory: 4Gi`) were initially too tight. The backend deployment (3 × 250m = 750m CPU) plus frontend (2 × 200m = 400m) plus monitoring (Prometheus 500m + Grafana 250m) totals 1.9 CPU, which is fine. But CronJobs that spawn outside the deployment's resource accounting can push past the quota. Increased to `requests.cpu: 6`.
- The `infrastructure/scripts/deploy.sh` wrapper uses `aws ecs update-service --force-new-deployment` which triggers a rolling update. But if the new image tag doesn't exist in ECR, ECS silently keeps running the old task definition. Added a pre-deployment check: `aws ecr describe-images --image-ids imageTag=$TAG` to fail fast on missing images.

## Recommendations
- Add a `terraform-validate` CI step that runs `terraform validate` and `terraform fmt -check` on every PR touching `infrastructure/terraform/`. Currently there's no automation — formatting drift accumulates between contributors.
- The Kubernetes manifests should use Kustomize overlays for environment-specific values (dev vs staging vs prod) instead of hardcoded values. Currently the manifests have `TODO: use kustomize` comments but no actual overlay structure.
- Consider adding Helm charts as an alternative to raw manifests. Helm would simplify version management and make it easier for external contributors to deploy their own instances.
