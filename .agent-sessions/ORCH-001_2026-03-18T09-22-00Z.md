## Summary
Implemented the `ChatSupervisor` class in `orchestration/src/supervisors/chat-supervisor.ts` as the central coordinator for the TypeScript chat orchestration layer. The supervisor classifies incoming user intents, routes them to the appropriate agent from the 16-agent registry, manages handoff chains (max depth 5), enforces daily cost budgets, and returns grounded responses. This also required wiring the `DualProviderClient` in `orchestration/src/llm/` to support Anthropic-primary / Google-fallback with automatic failover after 3 retries.

## Learnings
1. [KEEP] The `INTENT_CAPABILITY_MAP` in `agent-registry.ts` is the single routing table for all 16 agents. When adding a new agent, you must add its capabilities here AND register a Google fallback agent — otherwise failover silently returns `undefined` and the supervisor falls through to the generic assistant.
2. [KEEP] Anthropic's SDK throws `APIConnectionError` on network issues but `RateLimitError` on 429s — these need different retry strategies. Network errors should failover immediately to Google; rate limits should backoff-and-retry on the same provider first.
3. The `ContextManager` compaction algorithm keeps the system prompt + last N messages that fit within `maxTokens - reservedTokens`. It estimates tokens at 4 chars/token which is conservative but safe. A real tokenizer would be more accurate but adds a dependency.
4. Zod's `.passthrough()` on the environment schema caused silent acceptance of typos in env var names. Switched to `.strict()` so unknown keys throw at startup — caught two misspelled vars in local `.env` immediately.
5. The `CostTracker` daily reset uses `new Date().toISOString().slice(0, 10)` as the day key. This means UTC midnight resets, which is 7 PM EST — something to document for US-based operators who might see "budget exceeded" errors late in their workday when the UTC budget is actually exhausted.

## Gotchas
- The Anthropic SDK v0.39+ changed `messages.create()` to return a different shape when streaming is enabled. Had to check `stream` parameter and handle both `Message` and `Stream<MessageStreamEvent>` return types. Wasted ~45 minutes on type errors before finding the breaking change in their changelog.
- `pino` logger defaults to `stdout` which interferes with MCP stdio transport if both run in the same process. Had to configure the orchestration logger to write to `stderr` when `MCP_STDIO_MODE=true` is set.
- The Google GenerativeAI SDK does not surface rate-limit headers the same way as Anthropic. Had to detect rate limits by checking for `429` in the error message string rather than a typed error class. Fragile but no better option currently.

## Recommendations
- ORCH-002 should add streaming support to `DualProviderClient` — the current implementation only returns complete responses, which means the frontend cannot show partial text as it generates.
- The 16-agent registry is hardcoded in `createWithDefaults()`. Consider loading agent definitions from `agents/prompts/system/*.md` files at startup so adding a new agent doesn't require code changes, just a new markdown file.
- Integration tests should mock both providers at the SDK level using `msw` to verify failover behavior without hitting real APIs.
