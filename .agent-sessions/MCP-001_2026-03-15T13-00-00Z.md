## Summary
Migrated the MCP server from the old `ai_mcp_server/` package name to `mcp_server/` and implemented the full tool surface: 8 processing tools, 6 analysis tools, 6 operations tools, plus resources and prompts. The server uses FastMCP for stdio transport, wraps the `AgenticPipeline` from `agentic_ai/core/pipeline.py`, and includes an in-memory job store for async article processing. Added guardrails (content size limits, batch caps, job TTL, stderr-only logging) to prevent interference with the JSON-RPC stdio stream.

## Learnings
1. [KEEP] FastMCP uses stdio for JSON-RPC transport. ANY output to stdout from application code (print statements, debug logs, uncaught exception tracebacks) corrupts the JSON-RPC stream and crashes the Claude Code client. The `settings.py` forces all logging to stderr via `logging.StreamHandler(sys.stderr)` and the `ServerRuntime` sets `sys.stdout = sys.stderr` as a safety net during initialization.
2. [KEEP] The `process_article` tool creates an in-memory job and processes asynchronously. The job store is a plain dict, not persisted. If the MCP server restarts, all in-flight jobs are lost. For local development this is fine, but production deployments (Lambda, Azure Functions) are stateless by nature — the job store pattern doesn't translate. Consider Redis or DynamoDB-backed jobs for cloud.
3. [KEEP] The `.mcp.json` config at repo root defines the MCP server command as `PYTHONPATH=.. python -m mcp_server`. The `PYTHONPATH=..` is critical because `mcp_server/` imports from `agentic_ai/` using `from agentic_ai.core.pipeline import AgenticPipeline`. Without the path adjustment, Python cannot resolve the sibling package. This is a fragile coupling — if someone runs the MCP server from a different working directory, imports fail silently.
4. The `diagnose_provider_configuration()` tool checks for API keys by testing `os.environ.get(key)` truthiness, but does NOT attempt an actual API call. A key could be present but expired/invalid, and the diagnostic would still report "configured." Added a note in the tool description about this limitation.
5. MCP prompts support argument substitution via `{variable_name}` syntax in FastMCP. The 7 prompts use arguments like `{article_content}`, `{source_url}`, `{analysis_mode}`. If a caller omits a required argument, FastMCP returns a generic error without identifying which argument is missing. Added explicit argument validation in the prompt registration to surface better error messages.

## Gotchas
- The `purge_processing_jobs()` tool initially used `dict.pop()` in a loop while iterating, which raised `RuntimeError: dictionary changed size during iteration`. Fixed by collecting keys to delete first, then iterating the key list.
- The `compute_text_metrics()` tool calculates Flesch-Kincaid readability using `textstat` library. This is an optional dependency — if not installed, the tool returns a graceful error instead of crashing. But the `requirements.txt` in `agentic_ai/` did not list `textstat`. Added it.
- The rename from `ai_mcp_server` to `mcp_server` left stale references in `agentic_ai/Makefile` (the `run-mcp` target) and `docker-compose.yml`. Both needed updating. Easy to miss because neither file is imported by Python — they fail at runtime, not at import time.

## Recommendations
- Add integration tests that start the MCP server in a subprocess, send JSON-RPC requests via stdin, and verify structured responses on stdout. Currently there are no tests for the MCP surface — all validation is manual via Claude Code.
- The job store should have a background cleanup task that purges jobs older than `job_ttl_hours`. Currently jobs accumulate until the server restarts or `purge_processing_jobs()` is called manually.
- Consider splitting the 20 tools across multiple MCP servers (processing, analysis, operations) to reduce the tool surface per server. Claude Code performs better with fewer, focused tools than many broad ones.
