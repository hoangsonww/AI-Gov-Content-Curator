## Summary
Fixed a production-breaking issue where Pinecone API failures during vector search caused the entire sitewide chat endpoint to return 500 errors. The chat controller in `backend/src/controllers/chat.controller.ts` called `searchArticles()` from `pinecone.service.ts` without a try/catch, so any Pinecone timeout or rate limit would crash the request. Implemented graceful degradation: if vector search fails, the chat falls back to responding without article context (a "best-effort" reply with a disclaimer). Also hardened the crawler's `syncToPinecone()` path so embedding failures for individual articles don't abort the entire batch.

## Files Touched
- `backend/src/controllers/chat.controller.ts` — Wrapped `searchArticles()` call in try/catch. On failure, logs the error, sets `articleContext = []`, and appends a system note to the prompt: "Note: article search is temporarily unavailable. Respond based on your training data and note that sources could not be verified." This preserves chat availability even when Pinecone is down.
- `backend/src/services/pinecone.service.ts` — Added 3-second timeout to Pinecone `query()` calls. Added retry with 1s backoff for transient errors (connection reset, timeout). Added `isAvailable()` health check method that does a lightweight `describeIndexStats()` call. Changed `searchArticles()` to return `{ results: [], error: string }` instead of throwing, so callers can distinguish "no results" from "service down."
- `crawler/services/pinecone.service.ts` — Added per-article error handling in `syncBatchToPinecone()`. If embedding generation fails for one article (e.g. content too long for `gemini-embedding-001`), it logs the error and continues with the remaining articles instead of throwing. Added a summary log at the end: "Synced {N}/{total} articles to Pinecone, {failures} failed."
- `crawler/services/summarization.service.ts` — No code changes, but discovered during investigation that the Gemini API key rotation logic has a subtle bug: it rotates on *any* error, not just rate-limit errors. A malformed request will burn through all 4 keys before surfacing the actual error. Filed as a separate issue.

## Learnings
1. [KEEP] The `pinecone.service.ts` in `backend/` and `crawler/` are two separate files with duplicated Pinecone client initialization. They use the same env vars (`PINECONE_API_KEY`, `PINECONE_INDEX`) but have divergent error handling. Any fix to one must be checked against the other. Consider extracting a shared Pinecone utility package in the future.
2. [KEEP] The Pinecone JS SDK v3+ uses `@pinecone-database/pinecone` and its `query()` method does not have a built-in timeout. You must wrap it in a `Promise.race()` with a `setTimeout`-based rejection to enforce timeouts. The `fetch` API timeout option is not propagated through the SDK.
3. The `gemini-embedding-001` model has a 2048-token input limit. Articles longer than ~8000 characters can silently truncate or error depending on the SDK version. The crawler concatenates `title + " " + summary` for embedding input, which is usually safe, but articles with very long summaries (>6000 chars) can exceed the limit. Added a `substring(0, 7500)` guard.
4. Chat controller's `MAX_PROMPT_CHARS = 12000` includes both the article context and the user's message history. When Pinecone returns 5 long articles, the context alone can consume 10000+ chars, leaving only 2000 chars for history. This causes aggressive history truncation and the user feels like the bot "forgot" previous messages.

## Gotchas
- The Pinecone `describeIndexStats()` call used for health checking counts against the API rate limit. In high-traffic scenarios, the health check itself can trigger rate limiting. Reduced health check frequency to once per 60 seconds with in-memory caching of the last result.
- The backend `searchArticles()` return type change from `Article[]` to `{ results: Article[], error?: string }` required updating the chat controller, the orchestrated-chat controller, and two test files. Missing even one caller would cause a runtime type error with no compile-time warning because the backend uses loose TypeScript settings (`strict: false` in some areas).

## Recommendations
- Unify the two `pinecone.service.ts` files (backend and crawler) into a shared package or at minimum extract the client initialization and retry logic into a shared utility.
- Add a circuit breaker around Pinecone calls so that repeated failures don't keep hammering a downed service. The Python orchestration layer already has this pattern in `error_recovery.py` — port the approach to TypeScript.
- The Gemini key rotation bug in `crawler/services/summarization.service.ts` should be fixed in a separate bead (CRAWL-002 or similar). It wastes API keys on non-rate-limit errors.
