## Summary
Connected the TypeScript orchestration layer (`@synthoraai/orchestration`) to the backend Express API by implementing `backend/src/controllers/orchestrated-chat.controller.ts`. This controller creates a `ChatSupervisor` singleton, exposes endpoints for orchestrated chat (`/api/orchestrator/chat`, `/api/orchestrator/chat/stream`), article processing via the Python pipeline bridge (`/api/orchestrator/process`, `/api/orchestrator/batch`), and operational endpoints for health and cost snapshots. ## Learnings
1. [KEEP] The `ChatSupervisor` is instantiated as a singleton in the controller module scope. This means it shares state (context manager, cost tracker) across all requests in the same Node.js process. For Vercel serverless, each cold start gets a fresh instance — cost tracking resets on every cold start. For persistent servers (Docker, ECS), the singleton persists and cost tracking accumulates correctly. This behavioral difference between deployment targets is important to document.
2. [KEEP] The SSE streaming endpoint (`/api/orchestrator/chat/stream`) sets `Content-Type: text/event-stream`, `Cache-Control: no-cache`, and `Connection: keep-alive`. But Vercel's edge network buffers responses by default, which defeats SSE. Must set `X-Accel-Buffering: no` header for Nginx-based proxies and ensure Vercel route config has `streaming: true`. Without this, users see no output until the entire response is buffered, then get it all at once.
3. The `PipelineClient` health check calls `GET http://localhost:8100/health` on the Python FastAPI server. If the Python server isn't running, the health endpoint returns `{ python_pipeline: "unavailable" }` but the orchestrator still works for chat-only requests (TypeScript agents don't need the Python pipeline). This partial-health pattern is important — don't mark the entire orchestrator unhealthy just because one subsystem is down.
4. Zod schema validation at the controller level catches malformed requests before they reach the supervisor. But Zod error messages include the full schema path (e.g. `messages[0].content: Expected string, received number`) which can leak internal schema structure to clients. Added a `formatZodError()` helper that produces user-friendly messages without path details.

## Gotchas
- Express's `res.write()` for SSE does not automatically flush on all platforms. On Node.js 18+, `res.flushHeaders()` must be called after setting headers, and `res.flush()` (if using compression middleware) after each `res.write()`. Without this, the compression middleware buffers SSE events and delivers them in batches instead of individually. Spent 2 hours debugging "SSE works locally but batches in staging" before finding the compression interaction.
- The `handleBatchProcess` endpoint accepts an array of articles. Without a size limit, a malicious client could send thousands of articles and exhaust server memory. Added validation: max 25 articles per batch (matching MCP server's `max_batch_size`). Also added request body size limit of 5MB in the route config.
- The orchestration package uses `import` syntax (ESM) but the backend uses CommonJS `require()`. Had to configure the orchestration `tsconfig.json` to emit CommonJS (`module: "commonjs"`) and add `"type": "commonjs"` to its `package.json`. The alternative — converting the backend to ESM — would be a much larger change touching every import in the backend.

## Recommendations
- Add Playwright E2E tests for the streaming chat endpoint. Current tests only cover the non-streaming path because mocking SSE in Jest is awkward. Playwright can consume real SSE events from a running server.
- The `ORCHESTRATION_DAILY_BUDGET_USD` (default $10) should be surfaced in the frontend admin panel so operators can adjust it without redeploying. Currently it's only configurable via environment variable.
- Consider adding WebSocket support as an alternative to SSE. SSE is unidirectional (server → client), so the client cannot send follow-up messages on the same connection. WebSocket would enable true bidirectional streaming for multi-turn conversations.
