## Summary
Reviewed and hardened the newsletter service in `newsletters/`. The service generates daily digest emails containing the top articles from the past 24 hours, sends them via the Resend API, and tracks delivery status in MongoDB. Fixed an issue where the digest template was truncating article summaries at 100 characters (making them useless), increased to 300 characters. Also added unsubscribe link generation using signed tokens to prevent URL guessing, and added batch-send rate limiting to stay within Resend's API limits (100 emails/second on the free tier).

## Files Touched
- `newsletters/src/services/digest.service.ts` — Fixed summary truncation: changed `summary.substring(0, 100)` to `summary.substring(0, 300)`. Added `topArticles` sorting: now picks top 10 articles by a composite score of recency (last 24h), quality score (from pipeline), and topic diversity (no more than 3 articles per topic). Added `formatArticleForEmail()` helper that generates HTML card with title, source, summary snippet, topic badges, and read-more link.
- `newsletters/src/services/send.service.ts` — Added batch rate limiting: processes subscriber list in chunks of 50 with 600ms delay between chunks to stay under Resend's 100/second limit. Added per-subscriber error handling: if sending fails for one subscriber, logs the error and continues with the rest. Added delivery tracking: records `{ subscriberId, articleIds, sentAt, status }` in MongoDB `newsletter_deliveries` collection.
- `newsletters/src/services/subscriber.service.ts` — Added `generateUnsubscribeToken(subscriberId)` using `crypto.createHmac('sha256', UNSUBSCRIBE_SECRET).update(subscriberId).digest('hex')`. The unsubscribe link includes both the subscriber ID and the HMAC token. The `/unsubscribe` endpoint verifies the token before removing the subscriber, preventing enumeration attacks.
- `newsletters/src/templates/digest.html` — Updated email template: wider article cards (600px → full container width), larger summary text (14px → 16px), added source attribution with favicon, added topic badges with color-coding per topic category.
- `newsletters/pages/api/scheduled/sendNewsletter.ts` — Vercel cron handler. Added execution time logging, subscriber count logging, and a guard that skips sending if fewer than 3 articles are available (avoids sending empty-looking digests on slow news days).

## Learnings
1. [KEEP] The Resend API has different rate limits per plan: free tier is 100 emails/day and 1 email/second, the Pro tier is 50,000/month at 100/second. The batch chunk size and delay must be configurable via environment variables (`NEWSLETTER_BATCH_SIZE`, `NEWSLETTER_BATCH_DELAY_MS`) so operators can tune for their plan. The current hardcoded values (50 per chunk, 600ms delay) work for Pro tier but exceed the free tier's 1/second limit.
2. [KEEP] The newsletter Dockerfile (`newsletters/Dockerfile`) assumes a `dist/` directory exists from a prior TypeScript build step. But the current `package.json` build script runs `next build` (inherited from the frontend template), which generates `.next/` not `dist/`. This means the Docker image build fails. The fix is either: (a) add a `tsc` build step that outputs to `dist/`, or (b) change the Dockerfile to use `ts-node` or `tsx` at runtime. This is a known drift point documented in `CLAUDE.md`.
3. The `newsletter_deliveries` MongoDB collection grows by N×M records per day (N subscribers × M sends). For 1000 subscribers and 1 daily send, that's 365,000 records/year. Added a TTL index: `db.newsletter_deliveries.createIndex({ sentAt: 1 }, { expireAfterSeconds: 7776000 })` — 90-day retention. This must be created manually or via a migration script; the application code doesn't ensure the index exists.
4. The signed unsubscribe token prevents URL guessing but doesn't prevent replay. If a subscriber re-subscribes after unsubscribing, the old unsubscribe link still works because the token is based on the subscriber ID (which doesn't change). To prevent this, the token should include a subscription timestamp or nonce. Accepted this limitation for now since the risk is low (re-subscribing then getting unsubscribed via old link is annoying but not a security issue).

## Gotchas
- The Resend SDK's `emails.send()` method returns a `{ data, error }` tuple, not throwing on failure. Code that does `const result = await resend.emails.send(...)` without checking `result.error` silently drops failed sends. Every call site needs `if (result.error) { handle it }`.
- The Vercel cron schedule for the newsletter is defined in `vercel.json` as `"crons": [{ "path": "/api/scheduled/sendNewsletter", "schedule": "0 8 * * *" }]`. But the cron only fires in production deployments, not preview or development. During development, you must call the endpoint manually (`curl /api/scheduled/sendNewsletter`), which can accidentally send real emails if the `RESEND_API_KEY` is a production key. Added an `IS_DRY_RUN` environment variable check that logs the email content without sending.
- The email template uses inline CSS (required for email clients). The `<style>` tag approach doesn't work in Gmail, Outlook, or most webmail clients. Every style must be a `style=""` attribute on the HTML element. This makes the template verbose and hard to maintain. Consider using a CSS-to-inline tool like `juice` in the build pipeline.

## Recommendations
- Add a preview endpoint (`/api/newsletter/preview`) that renders the digest HTML for today's articles without sending, so editors can review content before the cron fires.
- The subscriber service should track bounce and complaint rates from Resend webhooks. Repeated hard bounces should auto-unsubscribe the address to maintain sender reputation.
- Consider offering topic-based digest preferences: subscribers choose which topic categories they care about, and the digest only includes matching articles. This requires a `preferences` field on the subscriber document and filtering logic in the digest service.
