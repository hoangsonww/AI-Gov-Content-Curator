## Summary
Fixed two issues in the crawler service. First, the Gemini API key rotation logic in `summarization.service.ts` was rotating keys on *any* error — including malformed requests and content-too-long errors — burning through all 4 configured keys before surfacing the actual problem. Refactored to only rotate on rate-limit (429) or quota-exceeded errors, and to surface non-rate-limit errors immediately. Second, improved the Puppeteer fallback logic in `crawler.service.ts`: the fallback from `networkidle2` to `domcontentloaded` was not waiting long enough for JS-heavy government sites, and the Chrome user-agent string was outdated (Chrome 112 → 124), causing some .gov sites to serve degraded content.

## Files Touched
- `crawler/services/summarization.service.ts` — Refactored `callGeminiWithRotation()`. Added `isRateLimitError(error)` helper that checks for HTTP 429, or error messages containing "quota", "rate", or "exceed" (case-insensitive). Only calls `rotateToNextKey()` when `isRateLimitError()` returns true. For non-rate-limit errors (400 bad request, 413 content too large, network errors), the error is thrown immediately without key rotation. Also added logging: `"Rotating API key due to rate limit on key index {i}"` vs `"Non-rate-limit error on key index {i}, surfacing immediately"`.
- `crawler/services/crawler.service.ts` — Updated `fetchDynamicArticle()`: increased `networkidle2` timeout from 15s to 20s, and the `domcontentloaded` fallback timeout from 10s to 15s. Added a 2-second `page.waitForTimeout()` after `domcontentloaded` to let late-loading JS hydrate. Updated Chrome user-agent from `Chrome/112.0.0.0` to `Chrome/124.0.0.0`. Added `page.setExtraHTTPHeaders({ 'Accept-Language': 'en-US,en;q=0.9' })` to prevent .gov sites from serving non-English content based on server-side language detection.
- `crawler/services/geminiModels.service.ts` — No code changes, but reviewed `getGeminiModels()` which returns the rotating model list. Confirmed it falls back to a hardcoded default list (`['gemini-2.0-flash-lite', 'gemini-1.5-flash']`) if the Gemini API model listing call fails. This is correct behavior.

## Learnings
1. [KEEP] The summarization service configures up to 4 Gemini API keys via `GOOGLE_AI_API_KEY`, `GOOGLE_AI_API_KEY_2`, `GOOGLE_AI_API_KEY_3`, `GOOGLE_AI_API_KEY_4`. Key rotation works by incrementing an index and wrapping around. But the rotation state is module-scoped (`let currentKeyIndex = 0`), meaning it resets to 0 on every Vercel cold start. In high-traffic scenarios where cold starts are rare, a rate-limited key stays rotated away for the lifetime of the process. In low-traffic scenarios with frequent cold starts, the same key (index 0) gets hit first every time and bears disproportionate load.
2. The crawler's HTML cleaning pipeline (`removeScriptTags`, `removeStyleTags`, `removeIframes`, `removeNoscript`, `removeLdJson`) runs after Cheerio parses the page. The order matters: removing `<script>` tags before parsing avoids Cheerio interpreting inline script content as text nodes. If cleanup runs post-parse, any `document.write()` output in scripts can leak into the extracted text.
3. Government sites (.gov) frequently use JavaScript frameworks (React, Angular) for content rendering. The Axios-first approach fails silently for these — it returns the page shell with `<div id="root"></div>` but no article content. The 403-detection heuristic (`if (response.status === 403) { fallback to Puppeteer }`) doesn't catch this case because the response is 200 OK. Added a content-length heuristic: if the extracted text is under 100 characters after cleanup, treat it as a JS-rendering failure and retry with Puppeteer.
4. [KEEP] Puppeteer's `page.goto()` with `waitUntil: 'networkidle2'` waits for fewer than 2 network connections for 500ms. On sites with analytics, tracking pixels, and ad networks, this can take a very long time or never resolve. The `domcontentloaded` fallback is faster but may miss dynamic content. The added 2-second wait after `domcontentloaded` is a pragmatic compromise for .gov sites that don't have heavy analytics but do hydrate React apps.

## Gotchas
- The `isRateLimitError()` check matches on error message substrings, which is fragile. The Gemini SDK doesn't expose a structured error code for rate limits — it throws a generic `GoogleGenerativeAIError` with a message string. If Google changes the error message wording, the detection breaks. Documented this fragility with a `// FRAGILE:` comment.
- Puppeteer's `page.setUserAgent()` must be called before `page.goto()`. The crawler was calling it after navigation, which means the first request used the default Puppeteer user-agent (headless Chrome). Moved the call to immediately after `browser.newPage()`.
- The `waitForTimeout()` method is deprecated in Puppeteer v22+ in favor of a custom delay function. Since the crawler pins Puppeteer at v21 in `package.json`, this works currently, but will break on upgrade. Added a `// TODO: replace with custom delay on Puppeteer upgrade` comment.

## Recommendations
- Add a content-quality check after extraction: if the article text is under 200 characters, or if it contains common JS-framework boilerplate strings (`"Loading..."`, `"Enable JavaScript"`, `"<noscript>"`), log a warning and optionally retry with Puppeteer. This catches more edge cases than the current 403-only heuristic.
- Consider implementing a headless browser pool (e.g. `generic-pool`) so multiple Puppeteer crawls can share browser instances. Currently each `fetchDynamicArticle()` call launches and closes a full Chrome instance. For batch crawls of 50+ articles, this creates significant startup overhead.
- The key rotation state should be persisted (Redis or even a simple file) so it survives cold starts and distributes load evenly across all 4 keys from the first request.
